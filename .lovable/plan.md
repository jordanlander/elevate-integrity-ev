I found a real deployment-path problem: this project has two GitHub Pages workflows that both deploy on pushes to `main`:

- `.github/workflows/deploy-pages.yml` builds with `SKIP_PRERENDER=true`
- `.github/workflows/jekyll-gh-pages.yml` also builds/deploys, but does not set `SKIP_PRERENDER`

That can create nondeterministic deploys where the last workflow to finish wins, which makes it look like changes are not reliably going live. Separately, the Lovable published URL is public, but it is a different deployment channel from GitHub Pages, so changes only appear there after using Lovable Publish.

Plan:

1. Remove the duplicate GitHub Pages deploy workflow
   - Keep one canonical workflow: `.github/workflows/deploy-pages.yml`
   - Delete `.github/workflows/jekyll-gh-pages.yml`
   - This prevents two different builds from racing to publish different outputs.

2. Add a deployment marker to the built site
   - Add a small generated marker file during GitHub Pages deploy, such as `dist/deploy-version.json`
   - Include the commit SHA and build timestamp
   - This gives you a concrete URL to verify exactly what commit is live.

3. Document the two deployment channels clearly
   - Update README so it says:
     - Lovable preview is not the same as the published site
     - Lovable published URL updates via Share -> Publish
     - GitHub Pages updates from pushes to `main`
     - The GitHub Pages workflow is the canonical GitHub deployment path

4. Keep architecture constraints intact
   - No Supabase changes
   - No form behavior changes
   - No visual redesign changes
   - No CallRail or tracking script changes

Validation after implementation:

- Confirm only one GitHub Pages workflow remains active
- Confirm the deploy marker is generated by the workflow
- Confirm no app behavior files are touched
